# 卡卡长期记忆、心情动作与轻主动陪伴设计

日期：2026-05-29

## 1. 目标

在现有「桌宠 + Cline 状态气泡 + DeepSeek 基础聊天」基础上，把卡卡升级为更像“长期陪伴生命体”的本地桌宠，重点解决以下问题：

1. 让卡卡能够长期记住用户是谁、喜欢什么、曾经聊过什么。
2. 让卡卡能够把旧对话和旧偏好聪明地找回来，而不是只依赖最近几轮上下文。
3. 让卡卡的动作/表情随心情、关系、活动状态变化，而不是只是固定状态贴图。
4. 在长期接入 DeepSeek API 的前提下，保持隐私优先：记忆长期保存在本地，只把最少必要上下文发给 DeepSeek。
5. 让卡卡具备低频、克制、但有生命感的主动行为，而不是频繁打扰用户。

## 2. 非目标

本阶段不直接做：

- 云端账户体系、多设备同步、远程数据库。
- 把所有长期记忆交给 DeepSeek 托管。
- Live2D、骨骼动画编辑器或复杂逐帧动作编排器。
- 开放式插件系统。
- 全自动高频主动提醒或系统通知轰炸。

## 3. 当前基础与约束

### 3.1 现有架构

项目当前已经具备：

- Electron 透明无边框桌宠窗口。
- Renderer 中的卡卡展示、对话气泡、临时聊天输入框。
- Main process 中的 DeepSeek 配置读取、聊天 IPC、Bridge 处理。
- MCP Server + 本地 HTTP Bridge，用于接收 Cline 工作状态。
- 12 状态资源包机制：`idle/happy/sleepy/thinking/angry/not-found/message/sleeping/head-pat/dragging/loading/signal-weak`。

### 3.2 素材现状

用户提供的 `D:\projects\cline-mcp-workspace\桌宠小人(1)\桌宠小人` 当前是一套 **12 张高分辨率单姿态 PNG**，语义上与现有 12 状态模型高度一致：

- 待机
- 开心
- 困困
- 思考
- 炸毛
- 装死 404
- 收到消息
- 睡觉
- 摸头反应
- 拖拽反应
- 加载中
- 信号弱

这意味着本阶段最合适的路线不是直接做复杂逐帧动画，而是先做 **心情/活动驱动的状态姿态切换系统**，并为后续更多素材预留扩展格式。

### 3.3 设计约束

- 长期记忆必须默认本地存储。
- DeepSeek API key 继续只保存在 main process / 本地配置。
- 发送给 DeepSeek 的信息必须是“当前消息 + 少量短上下文 + 少量相关记忆 + 当前心情摘要”，不能无边界转发所有历史。
- 行为层默认要克制，不得高频打扰。
- 资源包扩展要向后兼容现有 formatVersion 2 资源包。

## 4. 总体方案（推荐方案 A）

本阶段采用 **本地长期记忆优先 + DeepSeek 作为生成器 + 心情驱动姿态切换 + 低频主动陪伴** 的路线。

系统拆成 5 层：

1. **对话层（Conversation Layer）**
   - 接收用户输入。
   - 组织近期上下文。
   - 调用 DeepSeek 获取回复。

2. **记忆层（Memory Layer）**
   - 以本地存储为中心，维护用户档案、关系记忆、情景记忆。
   - 支持检索少量最相关长期记忆。

3. **心情层（Mood Layer）**
   - 根据最近互动、时间段、关系温度、记忆命中结果和系统状态计算当前心情。

4. **表现层（Presentation Layer）**
   - 将 mood/activity/bond 映射到卡卡当前姿态、表情和轻量表现。
   - 当前阶段主要驱动 12 状态姿态图切换。

5. **轻主动行为层（Presence Layer）**
   - 基于冷却规则、时间段、关系状态，在不打扰的前提下偶尔打招呼、问候或陪伴提示。

## 5. 长期记忆设计

长期记忆不做成“所有聊天记录都堆在一起”的大表，而是拆为三个本地模块。

### 5.1 用户档案记忆（Profile Memory）

存放长期稳定、与用户本人相关的认知：

- 用户称呼偏好。
- 喜欢/讨厌的内容。
- 常聊主题。
- 工作/作息习惯。
- 需要避免的说法或偏好边界。

适合结构化字段，例如：

```ts
type ProfileMemory = {
  displayName?: string;
  preferredAddress?: string;
  likes: string[];
  dislikes: string[];
  habits: string[];
  topics: string[];
  notes: string[];
  updatedAt: string;
};
```

### 5.2 关系记忆（Relationship Memory）

记录“卡卡与你之间发生过什么”以及关系演变：

- 熟悉度（familiarity）
- 亲密度（affection）
- 活跃度（engagement）
- 信任感（trust）
- 最近互动节奏
- 关键关系事件（例如被夸奖、重要节日、共识形成）

关系记忆不会直接全文喂给 DeepSeek，而是转成“当前关系状态摘要”。

### 5.3 情景记忆（Context Memory）

这是“上下文延续”的核心模块，负责把聊天历史分为三种层次：

1. **短期上下文**
   - 最近 N 轮对话原文。
   - 直接参与下一次聊天 prompt。

2. **中期摘要**
   - 把最近一段对话压缩成摘要。
   - 例如“最近在讨论桌宠动作系统与长期记忆设计”。

3. **长期片段**
   - 把可长期复用的信息存成带标签的记忆片段。
   - 例如“用户更偏好隐私优先”“用户希望卡卡像活着但不要太打扰”。

建议结构：

```ts
type ContextMemoryItem = {
  id: string;
  kind: "conversation-summary" | "fact" | "preference" | "project-context";
  text: string;
  tags: string[];
  sourceConversationId?: string;
  lastAccessedAt?: string;
  weight: number;
  createdAt: string;
  updatedAt: string;
};
```

## 6. 记忆检索策略

目标不是“把所有记忆都发给模型”，而是每次只挑少量最相关内容。

### 6.1 一期检索策略（推荐）

一期不强制引入向量数据库，先做 **本地轻检索**：

- 关键词命中
- 标签命中
- 最近访问加权
- 人工权重 `weight`
- 类别优先级（preference / fact / project-context）

每次聊天时从长期记忆中最多取：

- 1 条用户档案摘要
- 1 条关系状态摘要
- 1~3 条最相关情景片段

### 6.2 后续升级路径

后续若需要更聪明的检索，再增加：

- 本地嵌入向量缓存
- SQLite + FTS / vector-like 扩展
- 周期性记忆去重与摘要整理

但这些不作为本阶段首要实现范围。

## 7. 心情与关系状态机

### 7.1 状态输入

心情系统的输入来自四类信号：

1. **互动信号**
   - 用户是否刚聊天
   - 聊天语气偏正向还是挫败
   - 是否长时间未互动

2. **记忆信号**
   - 是否命中高价值长期记忆
   - 是否提到熟悉话题/偏好

3. **环境信号**
   - 时间段（白天/深夜）
   - 是否处于忙碌工作流（来自 Cline 状态）

4. **关系信号**
   - 当前亲密度/熟悉度/活跃度

### 7.2 一期 mood 集合（推荐）

建议先定义一层抽象 mood，而不是直接绑定 12 状态：

- `calm`：平静
- `happy`：开心
- `attached`：依恋 / 想陪伴
- `curious`：好奇 / 想知道更多
- `sleepy`：困困
- `upset`：炸毛 / 挫败
- `lonely`：安静地等你

### 7.3 relationship / bond 集合

建议用连续数值，而不是固定档位：

- `familiarity`: 0~100
- `affection`: 0~100
- `engagement`: 0~100
- `trust`: 0~100

然后在表现层里映射成：

- 陌生
- 熟悉
- 亲近

### 7.4 状态稳定规则

为了让卡卡“像活着”而不是“疯狂抖状态”，需要加入：

- mood 最短停留时间
- 切换冷却
- 某些情绪的衰减速度
- 用户工作忙时主动行为降频

## 8. 动作 / 表情 / 资源包扩展设计

### 8.1 当前阶段的实现方式

一期不做逐帧动作引擎，而是做：

**mood + activity + bond -> 表现姿态（Pet Status）**

现有 12 张 PNG 先作为第一版姿态池：

- `idle`：平静待机
- `happy`：开心 / 关系升温
- `sleepy` / `sleeping`：困困 / 夜间休息
- `thinking`：思考 / 检索记忆 / 回复中
- `angry` / `not-found`：炸毛 / 委屈 / 挫败
- `message`：收到消息 / 轻提醒
- `head-pat`：被正向互动后的亲密反馈
- `dragging`：被拖动时反应
- `loading`：忙碌状态
- `signal-weak`：异常或低能量感

### 8.2 资源包格式演进建议

当前 formatVersion 2 保持兼容；新增下一代扩展建议：

#### formatVersion 3（建议）

在保留 `states` 的同时增加可选扩展：

```json
{
  "id": "kaka-desktop-pet",
  "name": "卡卡桌宠小人",
  "version": "2.0.0",
  "formatVersion": 3,
  "states": {
    "idle": "idle.png",
    "happy": "happy.png"
  },
  "variants": {
    "idle": ["idle-soft.png", "idle-lookaway.png"],
    "happy": ["happy-soft.png", "happy-excited.png"]
  },
  "actionSets": {
    "greeting": ["message.png", "happy.png"],
    "memory-hit": ["thinking.png", "happy.png"]
  }
}
```

意义：

- 现在可以只用单张姿态图。
- 以后可逐步吃进更多素材。
- 如果未来你给出 PNG 序列帧，也可以在 `actionSets` 下再扩展 `frames[]`。

### 8.3 一期表现规则

为了不只是“瞬间切图”，加上轻量表达规则：

- 同一心情的姿态停留时间
- 重要记忆命中时优先触发更有感情的姿态
- 深夜默认更容易进入 `sleepy/sleeping`
- 关系更亲近时，相同聊天更容易触发 `happy` 或 `head-pat` 类表现

## 9. DeepSeek 长期开启的集成边界

DeepSeek 在系统中应扮演 **回复生成器** 和 **对话风格执行器**，而不是长期记忆主存储。

### 9.1 每次请求的 prompt 组成

建议结构：

1. **system prompt**
   - 卡卡人格边界
   - 回答风格
   - 不伪造能力

2. **relationship summary**
   - 当前关系温度、熟悉度、近期互动感觉

3. **profile summary**
   - 用户称呼/偏好/禁忌的简短摘要

4. **retrieved memories**
   - 1~3 条最相关长期记忆

5. **recent context**
   - 最近几轮聊天

6. **current user message**

### 9.2 记忆更新策略

每次对话后不立刻无脑写入所有内容，而是经过本地决策：

- 这一轮是否产生新事实？
- 是否更新用户偏好？
- 是否形成新关系事件？
- 是否需要生成中期摘要？

这样能避免记忆库迅速膨胀和污染。

## 10. 轻主动行为层设计

用户希望“别太打扰但要有点像活着”，因此主动行为层默认低频且带冷却。

### 10.1 一期主动行为类型

- 时间段问候
- 长时间未互动的轻陪伴
- 记忆命中后的“我记得你提过...”式回应

### 10.2 约束规则

- 有全局冷却时间
- 用户忙碌 / Cline 工作中时大幅降频
- 默认不用系统通知轰炸，只用桌宠气泡或轻提醒
- 不主动替用户做决定，不主动发长文本

### 10.3 后续扩展

后续可加：

- 节日/纪念日事件
- 心情低落时的更明显陪伴姿态
- 不同 bond 阶段的问候差异

## 11. 存储建议

### 11.1 一期推荐

先用本地 JSON / JSONL / 小型 SQLite 三者里最稳妥的一种。

推荐：

- `profile.json`
- `relationship.json`
- `context-memory.jsonl` 或 `context-memory.sqlite`

路径仍放在：

```text
%APPDATA%/cline-desktop-pet/
```

### 11.2 不推荐的一期做法

- 一开始就引入复杂向量数据库
- 一开始就依赖远程云记忆
- 直接把全部聊天历史原样塞进 prompt

## 12. 测试策略

### 12.1 记忆层测试

- 新事实写入
- 重复偏好去重/更新
- 摘要生成与检索排序
- 相关记忆命中策略

### 12.2 心情层测试

- 最近互动、长时间沉默、时间段、记忆命中对 mood 的影响
- mood 冷却和停留时间
- bond 数值更新

### 12.3 表现层测试

- mood/activity/bond -> status 映射
- 资源包在 formatVersion 2 / 3 下兼容
- 无扩展素材时 fallback 行为正确

### 12.4 主动行为层测试

- 冷却期阻止重复打招呼
- 忙碌状态下降低主动频率
- 长时间未互动时触发轻提醒

### 12.5 集成测试

- 聊天请求能携带相关记忆摘要
- DeepSeek 回复后记忆更新策略生效
- Cline 状态、心情状态、姿态切换协同工作

## 13. 分阶段落地建议

为了避免范围爆炸，建议正式实现分四个模块按顺序推进：

### 模块 A：本地长期记忆核心

- Profile / Relationship / Context 结构
- 本地存储
- 简单检索

### 模块 B：心情与关系状态机

- mood / bond 计算
- 冷却规则
- 聊天后状态更新

### 模块 C：动作/表情映射与资源包扩展

- mood/activity -> 12 状态姿态
- formatVersion 3 资源包草案
- variants / actionSets 预留

### 模块 D：轻主动行为层

- 时间段问候
- 低频陪伴
- 不打扰规则

## 14. 成功标准

如果本阶段设计与后续实现达成以下结果，就算成功：

1. 卡卡能在本地长期记住用户档案、关系状态和旧话题。
2. 聊天时能检索出相关旧记忆，而不是只靠最近几轮对话。
3. 卡卡表现不再只是固定状态切换，而是有“心情 / 活动 / 关系”驱动的姿态变化。
4. DeepSeek 长期开启时，仍保持本地记忆主导和隐私边界清晰。
5. 主动行为克制但有生命感，不会打扰用户。
6. 后续你继续提供动作素材时，不需要推翻架构，只需扩展资源包与映射规则。

## 15. 推荐的下一步

下一步不直接全量实现四个模块，而是先进入 implementation plan，按以下顺序规划：

1. 模块 A：本地长期记忆核心
2. 模块 B：心情与关系状态机
3. 模块 C：动作/表情映射与资源包扩展接口
4. 模块 D：轻主动行为层

这样能先把“记得你、能续聊、会变情绪”这个核心生命感做出来，再逐步增强动作资产利用率。
